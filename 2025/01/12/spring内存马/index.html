spring内存马

前置知识

Bean

通俗来讲：

  1. bean 是对象

  2. bean 被 IoC 容器管理

  3. Spring 应用主要是由一个个的 bean 构成的

ApplicationContext

Spring 框架中，BeanFactory 接口是 Spring IoC容器 的实际代表者。

ApplicationContext 接口继承了 BeanFactory,使用他是一个ioc容器

ContextLoaderListener 与 DispatcherServlet

我们先来了解一下，Root Context 和 Child Context :

  1. Spring 应用中可以同时有多个 Context，其中只有一个 Root Context，剩下的全是 Child Context
  2. 所有Child Context都可以访问在 Root Context中定义的 bean，但是Root Context无法访问Child Context中定义的 bean
  3. 所有的Context在创建后，都会被作为一个属性添加到了 ServletContext中

DispatcherServlet

DispatcherServlet 的主要作用是处理传入的web请求，根据配置的 URL pattern，将请求分发给正确的 Controller 和 View。DispatcherServlet 初始化完成后，会创建一个普通的 Child Context 实例。

依照规范，当没有显式配置 contextConfigLocation 时，程序会自动寻找 ``/WEB-INF/-servlet.xml，作为配置文件。因为上面的 是dispatcherServlet，所以当没有显式配置时，程序依然会自动找到 /WEB-INF/dispatcherServlet-servlet.xml` 配置文件。

综上，可以了解到：每个具体的 DispatcherServlet 创建的是一个 Child Context，代表一个独立的 IoC 容器；而 ContextLoaderListener 所创建的是一个 Root Context，代表全局唯一的一个公共 IoC 容器。

如果要访问和操作 bean ，一般要获得当前代码执行环境的IoC 容器 代表者 ApplicationContext

获取上下文

方法一：getCurrentWebApplicationContext

getCurrentWebApplicationContext 获得的是一个 XmlWebApplicationContext 实例类型的 Root WebApplicationContext。也可以是基于注解的 AnnotationConfigWebApplicationContext

方法二：WebApplicationContextUtils

通过这种方法获得的也是一个 Root WebApplicationContext

spring 5的WebApplicationContextUtils已经没有getWebApplicationContext方法

方法三：RequestContextUtils

通过这种方法获得的是一个 Child WebApplicationContext

它和方法二的差别就是少了WebApplicationContextUtils.getWebApplicationContext

我们来解释一下方法二和三

  1. (ServletRequestAttributes)RequestContextHolder.currentRequestAttributes():通过 RequestContextHolder 获取当前请求的属性
  2. getRequest():就是从当前请求的属性获取里面的请求HttpServletRequest
  3. RequestContextUtils.getWebApplicationContext:获取基于 HttpServletRequest 的 WebApplicationContext,也就是仅与当前请求相关的上下文信息
  4. WebApplicationContextUtils.getWebApplicationContext:从 WebApplicationContext 中获取 ServletContext，全局作用域的上下文也就是Root WebApplicationContext

方法四: getAttribute

所有的Context在创建后，都会被作为一个属性添加到了ServletContext中

查看当前所有的 attributes，发现确实保存有 Context 的属性名，

其中 org.springframework.web.servlet.DispatcherServlet.CONTEXT 和 org.springframework.web.servlet.DispatcherServlet.THEME_SOURCE 属性名中都存放着一个名叫 dispatcherServlet-servlet 的 Child WebApplicationContext,因此，我们用``dispatcherServlet-servlet这个Child WebApplicationContext`

上面代码中的 currentRequestAttributes() 替换成 getRequestAttributes() 也同样有效；getAttribute 参数中的 0代表从当前 request 中获取而不是从当前的 session 中获取属性值。

方法五:LiveBeansView

spring boot 初始化过程中会往 org.springframework.context.support.LiveBeansView 类的 applicationContexts 属性中添加 org.springframework.boot.web.servlet.context.AnnotationConfigServletWebServerApplicationContext 类的对象。

因为 org.springframework.context.support.LiveBeansView 类在 spring-context 3.2.x 版本（现在最新版本是 5.3.x）才加入其中，所以比较低版本的 spring 无法通过此方法获得 ApplicationContext 的实例。

获取HandlerMapping

DispatcherServlet 接收到客户端请求后，将其传递给 HandlerMapping。

HandlerMapping 根据配置（如注解、XML 配置等）解析请求，找到对应的处理器。

功能:

  1. 解析请求路径：根据客户端发来的 HTTP 请求 URI 和相关信息（如请求方法、参数等），匹配合适的处理器。

  2. 返回处理器: 如果匹配成功，HandlerMapping 返回一个 HandlerExecutionChain 对象，包含：处理器（Controller 方法或 Bean），与之相关的拦截器链（HandlerInterceptor)

Spring 2.5 开始到 Spring 3.1 之前一般使用 org.springframework.web.servlet.mvc.annotation.DefaultAnnotationHandlerMapping 映射器处理静态资源请求；

Spring 3.1 开始及以后一般开始使用新的 org.springframework.web.servlet.mvc.method.annotation.RequestMappingHandlerMapping 映射器来支持@Contoller和@RequestMapping注解。
